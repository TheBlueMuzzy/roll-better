<task type="auto">
  <name>Task 1: Add screen transitions</name>
  <files>src/App.tsx, src/App.css, src/components/MainMenu.tsx, src/components/WinnersScreen.tsx</files>
  <action>
1. Screen transition strategy: CSS opacity + transform transitions on the screen overlays. Each screen manages its own enter/exit animation via a CSS class toggle.

2. In MainMenu.tsx:
   - Add `visible` prop (boolean) — controls whether menu is shown
   - Use mount → rAF → add-class pattern for entrance animation (same as TipBanner):
     - On mount with visible=true: render with `.menu-backdrop` class
     - After rAF: add `.menu-visible` class which triggers CSS transition
     - On visible=false: remove `.menu-visible`, after transition end unmount
   - Simplification: keep it mounted when screen='menu', apply `.menu-visible` class. Transition handles opacity 0→1.

3. In WinnersScreen.tsx:
   - Same pattern: render when screen='winners', `.winners-visible` class triggers fade-in
   - Background opacity: use rgba(26, 26, 26, 0.92) so the 3D scene peeks through slightly

4. In App.tsx:
   - Game content (Canvas + HUD): wrap in a container div `.game-container` with opacity transition
   - When transitioning menu→game: menu fades out (0.3s), then game fades in (0.3s)
   - Implementation: use a simple state machine approach:
     - `screen` changes trigger CSS transitions via class names
     - Menu: always mounted but `.menu-visible` only when screen='menu'
     - Game container: `.game-visible` when screen='game' or screen='winners'
     - Winners: `.winners-visible` when screen='winners'
   - Delay game initialization until menu fade-out completes (add 300ms delay before calling initGame in handlePlay):
     ```
     // In handlePlay: fade menu out, then init game
     setScreen('game');
     setTimeout(() => {
       initGame(playerCount, difficulty);
       initRound();
       // spawn animation...
     }, 300);
     ```
     Actually simpler: just let CSS handle the fade. initGame can be called immediately since the menu overlay covers it during transition.

5. In App.css:
   - `.menu-backdrop` base: opacity 0, transition: opacity 0.3s ease
   - `.menu-backdrop.menu-visible`: opacity 1
   - `.winners-backdrop` base: opacity 0, transition: opacity 0.3s ease
   - `.winners-backdrop.winners-visible`: opacity 1
   - `.game-container`: opacity 0, transition: opacity 0.4s ease
   - `.game-container.game-visible`: opacity 1
   - Add `@keyframes slideUp` for winners rows if not already present (from 10-02)
   - Menu title entrance: slight translateY(-20px) → translateY(0) during fade-in for subtle lift effect
  </action>

<task type="auto">
  <name>Task 2: Persist game preferences + flow polish</name>
  <files>src/store/gameStore.ts, src/types/game.ts, src/components/MainMenu.tsx</files>
  <action>
1. In game.ts: Add `gamePrefs` to GameState:
   ```
   gamePrefs: {
     playerCount: number;   // last-used, default 3
     aiDifficulty: AIDifficulty;  // last-used, default 'medium'
   };
   ```

2. In gameStore.ts:
   - Add gamePrefs to initialState: `{ playerCount: 3, aiDifficulty: 'medium' }`
   - Add `setGamePrefs` action: `(prefs: Partial<GameState['gamePrefs']>) => void`
   - Implement: `setGamePrefs: (prefs) => set((s) => ({ gamePrefs: { ...s.gamePrefs, ...prefs } }))`
   - In `reset()`: preserve gamePrefs alongside settings (both persist across restarts)
   - Do NOT add localStorage persistence — Zustand in-memory is fine for now (session-scoped)

3. In MainMenu.tsx:
   - Read `gamePrefs` from store for initial selection state
   - Use gamePrefs.playerCount and gamePrefs.aiDifficulty as initial values for the local state selectors
   - When Play is pressed, update gamePrefs with current selections before calling onPlay:
     ```
     const handlePlay = () => {
       setGamePrefs({ playerCount, aiDifficulty: difficulty });
       onPlay(playerCount, difficulty);
     };
     ```
   - This way "Play Again" from winners screen uses the same settings without needing to pass them around

4. Flow polish in App.tsx:
   - handlePlayAgain: read gamePrefs from store instead of extracting from players array (cleaner):
     ```
     const handlePlayAgain = useCallback(() => {
       const { gamePrefs } = useGameStore.getState();
       initGame(gamePrefs.playerCount, gamePrefs.aiDifficulty);
       initRound();
       setScreen('game');
       // spawn animation...
     }, [initGame, initRound, setScreen, setPoolSpawning]);
     ```
   - Ensure handleMenu resets phase cleanly: `setPhase('lobby')` before `setScreen('menu')` so no stale sessionEnd phase lingers
  </action>
