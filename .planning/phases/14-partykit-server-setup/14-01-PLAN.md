**Key architectural decision (from ROADMAP):**
- Dice physics runs client-side (visual only). Server determines roll RESULTS (random numbers). Clients animate dice to show server-decided values.
- Free tier only: Partykit/Cloudflare 100k req/day.
- No accounts. 4-8 players. Host-authoritative validation.
- Partykit rooms are identified by room ID — connecting to a room ID creates it if it doesn't exist.

**Partykit project patterns (from Context7 research):**
- Server file implements `Party.Server` interface with `onConnect`, `onMessage`, `onClose` handlers
- `this.room.broadcast(message, [excludeId])` sends to all clients
- `this.room.getConnections()` iterates connected clients
- `Party.Room` has `id`, `storage` (async KV), `env` (environment vars)
- Client connects via `new PartySocket({ host, room })` from `partysocket` package
- Config via `partykit.json` at project root: `{ "name": "...", "main": "party/server.ts" }`
- Dev server: `npx partykit dev`
- Deploy: `npx partykit deploy`

**Phase 14 covers server setup only.** Phase 15 adds lobby UI + room codes. Phase 16 adds game state sync. Keep the message protocol focused on room lifecycle (join/leave/status), not game actions.

<task type="auto">
  <name>Task 2: Define shared message protocol types</name>
  <files>src/types/protocol.ts</files>
  <action>
  Create `src/types/protocol.ts` with typed message definitions for room lifecycle.

  **Design principles:**
  - Discriminated unions on `type` field for exhaustive switch statements
  - Separate client→server and server→client message types
  - Room lifecycle only (join/leave/state) — game action messages added in Phase 16
  - Player info is minimal: id, name, color (matches existing Player type subset)

  **Client → Server messages:**
  - `join` — player announces themselves: `{ type: "join", name: string, color: string }`
  - `leave` — graceful disconnect: `{ type: "leave" }`

  **Server → Client messages:**
  - `room_state` — full room snapshot (sent on connect + after any change): `{ type: "room_state", players: RoomPlayer[], hostId: string, status: RoomStatus }`
  - `player_joined` — delta update: `{ type: "player_joined", player: RoomPlayer }`
  - `player_left` — delta update: `{ type: "player_left", playerId: string }`
  - `error` — server error: `{ type: "error", message: string }`
  - `connected` — initial connection acknowledgment: `{ type: "connected", roomId: string, playerId: string }`

  **Supporting types:**
  - `RoomPlayer` — `{ id: string, name: string, color: string, isHost: boolean, isReady: boolean }` (isReady for Phase 15 lobby)
  - `RoomStatus` — `"waiting" | "playing" | "closed"` (simple enum for now)

  Export everything. Use `as const` satisfies pattern if helpful for type narrowing.
  Do NOT import from gameStore or any React code — this file must be importable by both server (party/) and client (src/).
  </action>
  <verify>
  - `npx tsc --noEmit -p tsconfig.server.json` passes (server can import protocol types)
  - `npx tsc --noEmit -p tsconfig.app.json` passes (client can import protocol types)
  - Types are discriminated unions (each message type has unique `type` string literal)
  </verify>
  <done>Protocol types defined. Both server and client tsconfigs can import them. Discriminated union on `type` field enables exhaustive switch handling.</done>
