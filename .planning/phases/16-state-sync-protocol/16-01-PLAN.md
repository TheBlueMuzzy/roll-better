  <name>Task 1: Game action protocol types</name>
  <files>src/types/protocol.ts</files>
  <action>
Add game action message types to the existing protocol file. Keep lobby messages intact — append new sections below them.

**Client → Server (add to ClientMessage union):**
- `RollRequestMessage`: `{ type: "roll_request" }` — player requests a roll (any player can trigger, first request starts the round's roll)
- `UnlockRequestMessage`: `{ type: "unlock_request", slotIndices: number[] }` — player's selected unlock slots
- `SkipUnlockMessage`: `{ type: "skip_unlock" }` — player skips unlock phase

**Server → Client (add to ServerMessage union):**
- `RollResultsMessage`: `{ type: "roll_results", playerResults: PlayerRollResult[] }` — roll results for ALL players in one message
  - `PlayerRollResult`: `{ playerId: string, rolled: number[], newLocks: LockedDieSync[], poolSize: number, lockedDice: LockedDieSync[] }`
  - `LockedDieSync`: `{ goalSlotIndex: number, value: number }` — same shape as game.ts LockedDie but defined in protocol to avoid importing game types
- `PhaseChangeMessage`: `{ type: "phase_change", phase: string }` — server tells all clients to transition (idle, rolling, locking, unlocking, scoring, roundEnd, sessionEnd)
- `RoundStartMessage`: `{ type: "round_start", round: number, goalValues: number[], players: PlayerSyncState[] }` — new round data
  - `PlayerSyncState`: `{ id: string, name: string, color: string, score: number, startingDice: number, poolSize: number, lockedDice: LockedDieSync[] }`
- `UnlockResultMessage`: `{ type: "unlock_result", playerId: string, unlockedSlots: number[], newPoolSize: number, lockedDice: LockedDieSync[] }` — one player's unlock applied
- `ScoringMessage`: `{ type: "scoring", winnerId: string, roundScore: number, players: PlayerSyncState[] }` — scoring results after a win
- `SessionEndMessage`: `{ type: "session_end", players: PlayerSyncState[] }` — game over

Keep the existing `GameStartingMessage` (used for lobby → game transition). The new messages handle in-game state sync.

Do NOT import from `game.ts` — protocol must stay import-free from game types (define `LockedDieSync` and `PlayerSyncState` locally in protocol.ts).

Add server-side game state tracking to the RollBetterServer class. Import `LockedDieSync` and `PlayerSyncState` from protocol.

**New server state (class properties):**
- `gameState`: `ServerGameState | null` — null during lobby, initialized on game start
- `ServerGameState` (define as interface in server.ts, NOT exported — server internal):
  ```
  interface ServerGameState {
    currentRound: number;
    goalValues: number[];
    phase: string; // idle, rolling, locking, unlocking, scoring, roundEnd
    players: ServerPlayerState[];
    rollRequestedBy: Set<string>; // track who requested roll
    unlockResponses: Map<string, { type: 'unlock' | 'skip', slotIndices?: number[] }>; // track unlock decisions
  }
  ```
- `ServerPlayerState` (define in server.ts):
  ```
  interface ServerPlayerState {
    id: string;
    name: string;
    color: string;
    score: number;
    startingDice: number;
    poolSize: number;
    lockedDice: LockedDieSync[];
    isOnline: boolean; // true = real player, false = AI bot
  }
  ```

**Initialize game state in handleStartGame:**
After the existing `game_starting` broadcast, initialize `this.gameState` from the lobby players. Online players get `isOnline: true`. Bot slots (targetPlayers > online count) get `isOnline: false`.

Call a new `serverInitRound()` method that:
1. Generates sorted goalValues (8 random 1-6, sorted ascending)
2. Resets all players: poolSize = startingDice, lockedDice = []
3. Sets phase to 'idle', increments currentRound
4. Broadcasts `round_start` message with round number, goalValues, and player states

**Note:** The `game_starting` message already has goalValues for the initial round. For the FIRST round, use those same values (don't generate twice). For subsequent rounds, serverInitRound generates new ones.

Add `import { findAutoLocks } from "../src/utils/matchDetection"` — this pure function works in any JS runtime, no React dependencies.

Also import `LockedDieSync` type from protocol.ts (it's the same shape as game.ts LockedDie but protocol-safe).

<task type="auto">
  <name>Task 3: Server roll handler</name>
  <files>party/server.ts</files>
  <action>
Implement `handleRollRequest` in the server's onMessage switch.

**Roll flow:**
1. Validate: game must exist (`this.gameState !== null`), phase must be 'idle', sender must be in the game
2. On first roll_request: set phase to 'rolling', then immediately generate results (no waiting for other players — simultaneous play means everyone rolls at once when anyone triggers it)
3. For EACH player (online + bots):
   - Generate `poolSize` random values (1-6 each)
   - Run `findAutoLocks(goalValues, rolledValues, player.lockedDice)` to compute new locks
   - Update server player state: add new locks to lockedDice, reduce poolSize by lock count
4. Build `PlayerRollResult[]` array with each player's rolled values, new locks, updated poolSize, and full lockedDice
5. Broadcast `roll_results` message to all clients
6. Set phase to 'locking'
7. After broadcasting roll_results, use `setTimeout(1000)` to advance phase:
   - Check if any player has 8 locked dice (winner) → handle scoring (Task in Plan 16-02)
   - Otherwise → set phase to 'unlocking', broadcast `phase_change`

**Important:** `findAutoLocks` import from `../src/utils/matchDetection` — this works because:
- It only imports `LockedDie` type from `../types/game` (type-only, erased at runtime)
- The function itself is pure JS logic, no React/browser deps
- Partykit compiles the server bundle and resolves imports

**Duplicate roll guard:** If phase is already 'rolling' or 'locking', ignore subsequent roll_request messages (prevent double-roll from network race).

Add `"roll_request"` case to the onMessage switch statement, calling `this.handleRollRequest(sender)`.
  </action>
