**Current round transition behavior:**
- Round end: goals exit → initRound (resets poolSize, clears locks) → goals enter → idle
- Pool dice just appear instantly when initRound sets new poolSize
- No exit animation for remaining pool dice

**Design direction (from CONTEXT.md):**
- Round End — Pool Exit: remaining pool dice scale up slightly (pop), then scale down to 0 and disappear
- Round Start — Pool Spawn: dice equal to startingDice total lerp FROM the Player Avatar group
  - Scale up from 0, overshoot slightly (bounce), scale back to correct size
  - WHILE scaling up, they are physically rolling (tumbling animation)
  - The roll result doesn't count — purely visual ("rolling themselves into existence")

**Exit animation concept:**
When a round ends (phase transitions to 'scoring' or 'roundEnd'), the human player's remaining pool dice should:
1. Pop up slightly (scale 1 → ~1.3 over ~0.15s, ease-out).
2. Then scale down to 0 (over ~0.3s, ease-in).
3. Disappear.

**Implementation approach:**
- Add a `poolExiting` boolean to RoundState (or a separate state field).
- When round ends (phase = 'roundEnd'), set poolExiting = true.
- DicePool reads poolExiting. When true, each PhysicsDie (or a wrapper) plays the pop+shrink animation.
- Use a simple useFrame animation on each die's scale: phase 1 (0–0.15s) scale up with ease-out, phase 2 (0.15–0.45s) scale down with ease-in.
- After animation completes (~0.5s), the dice are at scale 0 — they're visually gone.
- initRound (which resets poolSize) should happen AFTER the exit animation finishes.

**Timing adjustment (App.tsx):**
Current roundEnd sequence: exit goals (0ms) → initRound (500ms) → enter goals → idle (1500ms).
New sequence:
- 0ms: poolExiting = true (pool dice start pop+shrink)
- 0ms: setGoalTransition('exiting') (goals slide right) — runs in parallel with pool exit
- 500ms: initRound() — pool dice gone by now, new pool will spawn with spawn animation
- 500ms: setGoalTransition('entering')
- 1500ms: idle

The pool exit animation (0.45s) fits within the existing 500ms window before initRound. Clear poolExiting when initRound is called.

**AI pool dice:** AI don't have visible pool dice (they roll behind the scenes), so no exit animation needed for AI. Only the human player's pool dice exit.

**Spawn animation concept:**
When a new round starts (initRound creates fresh pool), the human player's pool dice should:
1. Start at the Player Avatar group position, scale 0.
2. Fly to their pool positions (scattered in rolling area).
3. Scale up from 0 with overshoot (bounce: 0 → 1.15 → 1.0).
4. During flight, tumble/rotate continuously (visual rolling, not physics).
5. Once at destination and at full scale, become regular PhysicsDie instances (physics takes over).

**Implementation approach — SpawningDie component:**
Create a temporary animation that plays BEFORE PhysicsDie mounts:
- Add `poolSpawning` boolean to RoundState.
- When initRound is called, set poolSpawning = true.
- Scene renders SpawningDie instances (one per startingDice) instead of DicePool while poolSpawning is true.
- Each SpawningDie:
  - fromPos: player profile group position [profileX, 0, playerRowZ]
  - toPos: a random position in the rolling area (use existing DicePool spawn logic or generateSpawnPosition)
  - Scale: ease-out with overshoot. Use `1 + 0.15 * sin(t * PI)` for the bounce (peaks at 1.15 midway, settles to 1.0).
  - Rotation: continuous tumble during flight (rotX += delta * 10, rotZ += delta * 7).
  - Duration: ~0.6s per die.
  - Stagger: ~0.08s per die (rapid succession from avatar).
- After all spawn animations complete, set poolSpawning = false.
- DicePool then renders normally with PhysicsDie instances at the spawn positions.

**Alternative simpler approach:**
Instead of a separate SpawningDie, modify DicePool/PhysicsDie to play a scale-in animation on mount:
- PhysicsDie reads a `spawning` prop.
- When spawning, start at scale 0, animate to 1 with overshoot.
- Apply tumble rotation during scale-up.
- Once at full scale, enable physics (currently physics bodies exist on mount — may need to delay gravity or use kinematic→dynamic switch).

Choose whichever approach is cleaner. The SpawningDie approach (animation component → swap to PhysicsDie) is more aligned with existing patterns (AnimatingDie for locks, MitosisDie for unlocks).

**Timing:**
Spawn animation starts when initRound is called (500ms into roundEnd sequence). It should complete before the player can roll (idle phase at 1500ms). 0.6s + stagger for 2-12 dice = ~0.6 + 0.08*12 = ~1.56s max. May need to extend the roundEnd→idle window slightly for large pool sizes, or tighten stagger for many dice.

**AI pool spawn:** Not needed — AI dice are invisible (rolled as random numbers). Only human player sees spawn animation.
