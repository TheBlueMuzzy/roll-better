  <files>src/hooks/useRoom.ts, src/hooks/useOnlineGame.ts, src/components/HUD.tsx, src/App.tsx</files>
  <action>
**Fix reconnect toast (broken — event never dispatched):**

1. In useRoom.ts, the `player_reconnected` message case exists but only passes to useOnlineGame. In useOnlineGame.ts, the `player_reconnected` handler should dispatch the custom event that App.tsx is listening for:
   - Find where `player_reconnected` is handled in useOnlineGame.ts
   - Add: `window.dispatchEvent(new CustomEvent('player-reconnected', { detail: { name: msg.name || 'A player' } }))`
   - App.tsx already listens for this event (line ~96) and shows the toast — just needs the dispatch

2. If `player_reconnected` message doesn't include a player name field, check protocol.ts — add `name?: string` to the message if missing, and have the server include the reconnecting player's name when broadcasting.

**Fix seat state notification text accuracy:**

3. In HUD.tsx (line 51-59), the seat state change notifications need context-aware text:
   - `bot → human-active`: Currently "X reclaimed their seat" — this is wrong when it's a NEW player taking over. But the client can't easily distinguish reclaim vs new-join from the seat_state_changed message alone.
   - Solution: Add a `reason?: 'reclaim' | 'takeover'` field to the `seat_state_changed` or `seat_takeover` server message so the client can show:
     - Reclaim (same persistentId): "X is back!"
     - Takeover (new player): "X joined the game"
   - Update protocol.ts SeatTakeoverMessage (or SeatStateChangedMessage) to include `reason` field
   - Update server `executePendingSeatClaims` to set reason based on whether joiner's persistentId matches the bot's original persistentId
   - Update HUD.tsx to use the reason for appropriate text

4. Verify the `human-active → human-afk` notification ("X is on autopilot") and `human-afk → bot` ("Bot took over for X") are still accurate — these should be fine, just confirm.

Do NOT change the reconnect overlay in App.tsx ("Reconnecting..." spinner) — that works correctly.
  </action>

<task type="auto">
  <name>Task 2: Mid-game join waiting status improvements</name>
  <files>src/components/MainMenu.tsx, src/App.css</files>
  <action>
After claiming a seat mid-game, the client shows "Taking over next round..." (or "Reclaiming your seat..."). This is functional but could be clearer.

1. In MainMenu.tsx, when `claimedSeat` is set (seat claimed, waiting for phase boundary):
   - Show a pulsing dot or spinner next to the status text to indicate active waiting
   - Change text to be more specific:
     - If `autoMatched`: "Reclaiming your seat... Joining next round"
     - If not autoMatched: "Seat claimed! Joining next round..."
   - Add a subtle CSS animation (reuse the connection-spinner pattern or a simple pulse)

2. When `seatClaimError` is shown, make sure the error text is clear:
   - `"seat_taken"` → "That seat was just taken — pick another"
   - `"not_a_bot"` → "That seat is no longer available"
   - Default: Show the raw reason text

3. Add a "CANCEL" option while waiting for takeover — clicking it sends no message (server handles disconnect cleanup) but returns client to the seat selection view by clearing `claimedSeat` state. This helps if user claimed wrong seat.
   - In useRoom.ts, add a `cancelClaim()` function that clears `claimedSeat` and `seatClaimError`
   - Note: server still has the pending claim — but that's fine, when user claims a different seat, the server's handleSeatClaim replaces the old claim

Do NOT add a countdown timer — we don't know when the next phase boundary will occur, so a timer would be misleading. The spinner/pulse is sufficient.
  </action>
