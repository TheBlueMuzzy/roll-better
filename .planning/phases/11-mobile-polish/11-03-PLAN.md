<task type="auto">
  <name>Task 1: Viewport + touch + Safari hardening</name>
  <files>index.html, src/App.css</files>
  <action>
1. **index.html — viewport meta:**
   - Update viewport meta to: `width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover`
   - `viewport-fit=cover` enables safe-area-inset access for notch devices
   - Add: `<meta name="apple-mobile-web-app-capable" content="yes">`
   - Add: `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`

2. **src/App.css — body/root hardening:**
   Add to html/body rules (create if needed):
   - `position: fixed; width: 100%; height: 100%; overflow: hidden;` — prevents iOS rubber-band scrolling
   - `overscroll-behavior: none;` — prevents pull-to-refresh on Chrome Android
   - `touch-action: manipulation;` — kills 300ms tap delay, prevents double-tap-to-zoom
   - `-webkit-tap-highlight-color: transparent;` — removes blue flash on tap (iOS/Android)
   - `user-select: none;` — prevents text selection during gameplay (game content only, not inputs)
   - `-webkit-touch-callout: none;` — prevents long-press popup on iOS

3. **Safe area insets:**
   - On the `#root` or `.game-container` element, add:
     `padding: env(safe-area-inset-top) env(safe-area-inset-right) env(safe-area-inset-bottom) env(safe-area-inset-left);`
   - This ensures content doesn't hide behind notch/home indicator
   - Apply to HUD elements that sit at screen edges (hud-top, hud-bottom, settings-gear)

4. **Height fix:**
   - Use `height: 100dvh` on root container (dynamic viewport height — accounts for mobile browser chrome appearing/disappearing)
   - Fallback: `height: 100vh` for browsers without dvh support
   - Pattern: `height: 100vh; height: 100dvh;` (cascade fallback)

Do NOT add `touch-action: none` (that breaks all touch interactions). Use `manipulation` which only kills zoom gestures while preserving pan and tap.
  </action>
  <verify>Build succeeds. No visible layout changes on desktop. Mobile: no zoom on double-tap, no pull-to-refresh, no rubber-banding, no tap highlight flash.</verify>
  <done>Viewport hardened, touch behavior locked down, safe areas respected, 100dvh height, iOS rubber-banding prevented</done>
</task>

<task type="auto">
  <name>Task 2: Wire performanceMode to Canvas rendering optimizations</name>
  <files>src/App.tsx, src/components/Scene.tsx</files>
  <action>
1. **App.tsx — Canvas dpr and shadows:**
   - Read `performanceMode` from store
   - Set Canvas `dpr` prop: `performanceMode === 'simple' ? 1 : [1, 2]`
     - Simple: force 1x pixel ratio (biggest single performance win on high-DPI phones)
     - Advanced: let R3F pick between 1x-2x based on device
   - Set Canvas `shadows` prop: `performanceMode === 'advanced'` (disable shadows entirely in simple mode)

2. **Scene.tsx — conditional AccumulativeShadows:**
   - Accept `performanceMode` as prop (or read from store)
   - In simple mode: do NOT render AccumulativeShadows component at all (conditional render `{performanceMode === 'advanced' && <AccumulativeShadows ... />}`)
   - In simple mode: use a simpler ground plane (just the dark wood MeshStandardMaterial without shadow receiving, or a basic shadow beneath dice using a simple circular shadow plane if needed for visual grounding)
   - Keep lighting the same in both modes — removing the spotlight would change the look too much

3. **Performance mode label update:**
   - In Settings.tsx, update the hint text to be more descriptive:
     - Advanced: "Best visuals" (was just "Advanced")
     - Simple: "Better battery" (was just "Simple")
   - This helps users understand the tradeoff

Do NOT downgrade dice materials in simple mode — the premium dice look is core to the product. Only reduce environmental rendering (shadows, dpr). The dice roll IS the product.
  </action>
