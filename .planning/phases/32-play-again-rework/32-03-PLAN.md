**MainMenu.tsx — auto-detect lobby return:**
Add a useEffect that detects returning to lobby:
- When `room.isConnected && room.status === 'waiting' && room.players.length > 0 && onlineMode === 'idle'`:
  - Set `onlineMode` to `'creating'` if `room.isHost`, or `'joined'` if not
  - This triggers the lobby view (room code, player list, Start/Ready buttons)
- This covers the case where screen changes from 'winners' to 'menu' while the room connection is already active in "waiting" status

**MainMenu.tsx — Play Again lobby shows all players correctly:**
The existing lobby UI shows player list with ready indicators. In the post-game lobby:
- Players who clicked Play Again are ready (green check)
- Players still on winners screen are not ready (red X) — they're still in the server's players map
- Host sees Start button and can start anytime (existing behavior)

Verify: the existing lobby UI naturally handles this. The ready indicators use `player.isReady`. No changes needed if server sends correct ready states.

**MainMenu.tsx — handle the "claiming" transition for late Play Again:**
When `play_again_ack` has mode `"mid_game_join"`, the server follows up with `seat_list`. The existing claiming mode detection (when seatList becomes non-null) should trigger automatically. Verify this works:
- onlineMode should transition to `'claiming'` when seatList arrives
- The seat selection UI renders (existing from Phase 30)

Check the existing seatList detection logic in MainMenu. It currently detects mid-game join in the 'joining' mode. For late Play Again, the player might be in 'idle' mode (just returned from winners). Add detection: if seatList becomes non-null and onlineMode is 'idle', transition to 'claiming'.
