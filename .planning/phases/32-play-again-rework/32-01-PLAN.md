**Protocol additions (protocol.ts):**
- Add `PlayAgainMessage` to ClientMessage union: `{ type: "play_again" }`
- Add `PlayAgainAckMessage` to ServerMessage union: `{ type: "play_again_ack"; mode: "lobby" | "mid_game_join" }` — tells client which UI to show

**Server state additions (party/server.ts):**
- Add class property: `previousGamePersistentIds: Map<string, number> = new Map()` — maps persistentId → seatIndex from last game
- Add class property: `previousGameTargetPlayers: number = 0` — previous game's player count (for bot fill)

**handlePlayAgain method (party/server.ts):**
Route `play_again` in the onMessage switch to this new method.

Logic for when `this.gameState?.phase === "sessionEnd"` (first transition):
1. Save previous game context: iterate `this.gameState.players`, build `previousGamePersistentIds` map (persistentId → seatIndex), save `previousGameTargetPlayers = this.gameState.players.length`
2. Call `this.cleanupTimers()` — clears mid-game joiners, pending claims, all game timers
3. Set `this.gameState = null`
4. Set `this.status = "waiting"`
5. For ALL players in `this.players`: set `isReady = false`
6. Set the sender's `isReady = true`
7. Send `play_again_ack { mode: "lobby" }` to sender only
8. Broadcast `room_state` to all connections (existing broadcastRoomState or equivalent)

Logic for when `this.status === "waiting"` (subsequent play_again from another player):
1. Set sender's `isReady = true`
2. Send `play_again_ack { mode: "lobby" }` to sender
3. Broadcast `room_state`

Logic for when `this.status === "playing"` and phase is NOT sessionEnd (game already started — late joiner):
- This case is handled in Plan 32-02. For now, just send `play_again_ack { mode: "mid_game_join" }` and defer to Plan 32-02 for the full routing logic.

Guard: If player is not in `this.players`, ignore (they already left or never joined).

**Remove old restart_game handling:**
- Delete `handleRestartGame` method entirely — it's replaced by the play_again → lobby → start_game flow
- Remove the `restart_game` case from onMessage switch

**Modify handleStartGame to handle unready players:**

Current behavior: rejects start if any non-host player is NOT ready.
New behavior: start anyway — unready players become bot seats.

Changes to handleStartGame:
1. Remove or modify the "all non-host must be ready" validation. Instead: proceed even if some players aren't ready.
2. Before building the game, separate players into ready vs unready:
   - Ready players (isReady === true OR isHost): included as human players
   - Unready players: remove from `this.players` map (their WebSocket stays open — they can join later via mid-game join)
3. Use `this.previousGameTargetPlayers` as the target player count if > 0 (otherwise fall back to current players + enough for a full game). This ensures the same number of seats as the previous game.
4. Build game state: ready players as humans + bots to fill remaining seats up to targetPlayers
5. Rest of handleStartGame stays the same (generate goals, broadcast game_starting, create gameState, serverInitRound)

**Important:** When removing unready players from `this.players`, do NOT close their connection. They remain connected but untracked. If they later send `play_again`, Plan 32-02's logic will route them to mid-game join.
