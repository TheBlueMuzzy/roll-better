**PhysicsDie.tsx — Add rotational tumble on release:**
In the `setAttractTarget(null)` branch (line 151-166), after restoring gravity and colliders, apply a random torque impulse for tumble feel. The dice already have linear velocity from orbital tracking (setLinvel), so they have momentum. Add:
```
body.applyTorqueImpulse(
  { x: randRange(-2, 2), y: randRange(-2, 2), z: randRange(-2, 2) },
  true
);
```
This matches the torque range used in `roll()` (line 119-122). Do NOT zero out linvel — the existing orbital velocity IS the roll momentum.

**Scene.tsx — Release triggers roll:**
In `handleFloorPointerUp()` (line 224-228), after calling `stopGathering()`:
1. Call `onRoll?.()` to trigger the roll pipeline (sets phase='rolling' via App.tsx)
2. Do NOT call `dicePoolRef.current?.rollAll()` — dice are already in motion from gather release. The `rollAll()` path teleports dice and applies fresh impulses, which would fight the orbital momentum.

The key insight: gather-release IS the roll. Dice have momentum + tumble from the attractor. We just need to transition to 'rolling' phase so the settle→results pipeline fires.

**Scene.tsx — Remove tap-to-roll:**
Remove or disable `handleFloorClick()` (line 230-235). The gather pointer events (down→move→up) replace tap-to-roll entirely. Remove the `onClick` prop from RollingArea floor mesh that triggers this.

**App.tsx — Split handleRoll for gather path:**
`handleRoll` (line 501-514) currently calls both `setPhase('rolling')` AND `sceneRef.rollAll()`. For the gather-release path, we only need `setPhase('rolling')` — the dice are already moving. Two options:
- Add a parameter: `handleRoll(skipRollAll?: boolean)` — gather release passes `true`, AFK fallback passes nothing
- Or: Create `handleRollStart` that only sets phase (already exists at line 516-518!) — use this for gather release, keep `handleRoll` for AFK fallback

Use the existing `handleRollStart` callback for gather-release path. It already does `setPhase('rolling')` (line 517). Wire Scene's `handleFloorPointerUp` to call `onRollStart?.()` instead of `onRoll?.()`.

**Audio init:** The first gather-release also needs to init audio (autoplay policy). Add the audio init check from `handleRoll` (lines 504-509) into `handleRollStart` as well.

**AFK mid-gather timeout — force release:**
If the AFK timer fires while the player is mid-gather (phase='gathering' or gatherState.active), we need to force-release:
1. In HUD.tsx `handleIdleTimeout`, check if `useGameStore.getState().gatherState.active` is true
2. If active: call a new `onForceRelease?.()` callback that triggers the gather-release path (stopGathering + onRollStart)
3. If not active (normal idle): call `onRoll()` as before (rollAll path)

Wire `onForceRelease` from HUD through App.tsx to Scene.tsx. In Scene.tsx, add a `forceRelease()` method on SceneHandle (via useImperativeHandle):
```
forceRelease() {
  if (useGameStore.getState().gatherState.active) {

  // Dice are already in orbit — release them
  dicePoolRef.current?.releaseAll(); // sets attractTarget(null) on all dice
  onRollStart?.();
}
```

Add `releaseAll()` to DicePool that loops all dieRefs calling `setAttractTarget(null)`. This is the same loop that already runs in the useFrame when gatherActive goes false — but we need an imperative version for the forced path.
