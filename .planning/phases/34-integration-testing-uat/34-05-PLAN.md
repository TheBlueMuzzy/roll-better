<task type="checkpoint:human-verify" gate="blocking">
  <what-built>Duplicate connection detection (Phase 33) and AFK escalation (Phase 28) ready for testing.</what-built>
  <how-to-verify>
    ### Scenario A: Multi-tab duplicate connection
    1. Open the game in Tab 1, create a room, join a game
    2. Open the game in Tab 2 (SAME browser — same localStorage = same persistentId)
    3. In Tab 2, enter the same room code and try to join
    4. Confirm: Tab 1 receives "connected in another tab" error and shows BACK TO MENU
    5. Confirm: Tab 2 connects successfully (new connection replaces old)
    6. Confirm: Game state in Tab 2 is correct (if Tab 1 was mid-game)

    ### Scenario B: AFK escalation — 2 timeouts → bot takeover
    1. Start a 2-player online game (2 browser windows)
    2. On one player's window, do NOT interact — let the rolling timer expire
    3. Confirm: Server auto-rolls for AFK player (bot behavior for 1 turn)
    4. Confirm: AFK player's avatar shows bot icon or AFK indicator
    5. Let a SECOND consecutive timeout happen (don't interact again)
    6. Confirm: After 2 consecutive AFK turns, player is promoted to full bot
    7. Confirm: The AFK player's browser shows "You've been replaced by a bot" or similar, returns to menu
    8. Confirm: Other player sees the seat fully transitioned to bot

    ### Scenario C: AFK counter reset on manual action
    1. Start fresh game, let 1 timeout happen (AFK counter = 1)
    2. On the NEXT turn, manually roll (interact normally)
    3. Confirm: AFK counter resets — player is no longer at risk
    4. Let another single timeout happen
    5. Confirm: Player is NOT promoted to bot (counter was reset, only at 1 again)

    Report which scenarios passed/failed.
  </how-to-verify>
