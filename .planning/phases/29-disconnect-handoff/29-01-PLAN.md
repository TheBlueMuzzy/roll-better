<task type="auto">
  <name>Task 1: Replace 60s keepalive with per-player disconnect grace timer</name>
  <files>party/server.ts</files>
  <action>
  Rework `removePlayer()` to start a per-player grace timer instead of the room-level 60s keepalive:

  1. **Add a `disconnectGraceTimer` Map** (connId -> setTimeout handle) as a class property alongside existing timer properties.

  2. **In `removePlayer()`** (currently lines ~1359-1435):
     - Still mark `gamePlayer.isOnline = false` and handle intentional leave as before.
     - Instead of the 60s keepalive logic at the bottom, start a per-player grace timer:
       - **If game is in a timed phase** (rolling or unlocking — check `gameState.phase`): grace duration = time remaining on the current phase timer. Calculate from when the phase timer was set. Store the phase timer start time in a new property (e.g., `phaseTimerStartedAt: number | null`) that gets set whenever `rollingTimeoutTimer` or `unlockTimeoutTimer` is started.
       - **If game is in a non-timed phase** (locking, scoring, roundEnd, session_end): grace = 0, promote to bot immediately. These are transient phases where waiting makes no sense.
       - **If game is in lobby/not started**: just remove the player from the room (existing behavior, no grace needed).
     - When grace timer fires: call `promoteToBotFromAFK(player)` + `broadcastSeatStateChanged(...)`. The player was already marked offline — this just transitions their seat to bot.
     - Broadcast `seat_state_changed` with `newState: 'bot'` so all clients update.

  3. **Keep room-level keepalive for empty rooms**: If ALL human players are now offline/bot after a disconnect (no connected humans left), start a short keepalive (10s) to allow quick reconnects. If a human reconnects, cancel it. If it fires, close the room. This replaces the old 60s timer.

  4. **Track phase timer start times**: When setting `rollingTimeoutTimer` (line ~687-690) or `unlockTimeoutTimer` (line ~1028-1031), record `this.phaseTimerStartedAt = Date.now()` and `this.phaseTimerDuration = 25000` (the timeout duration). Clear these when timers are cleared.

  5. **Clean up grace timers**: Add cleanup to `cleanupTimers()` — iterate and clear all entries in `disconnectGraceTimer` Map.

  Do NOT change the AFK escalation logic (autoRollUnresponsivePlayers / autoSkipUnresponsivePlayers) — those handle connected-but-idle players. This task handles disconnected players only.

  <name>Task 2: Handle reconnect during grace window</name>
  <files>party/server.ts</files>
  <action>
  Update `onConnect()` reconnection logic (currently lines ~97-200) to cancel grace timers and restore seat state:

  1. **Cancel grace timer on reconnect**: When a reconnecting player is found in `gameState.players` (matching conn.id, !isOnline, !intentionalLeave), check `disconnectGraceTimer` for their connId. If found, `clearTimeout` and delete the entry.

  2. **Restore seat state**: After canceling the grace timer, restore the player's seat:
     - Set `gamePlayer.seatState = 'human-active'`
     - Reset `gamePlayer.autopilotCounter = 0` (they came back — fresh start)
     - Broadcast `seat_state_changed` with `newState: 'human-active'` so all clients see the player return

  3. **Cancel empty-room keepalive**: If the room keepalive was running (all humans were offline), cancel it since a human just reconnected.

  4. **Edge case — reconnect after grace expired**: If the grace timer already fired (player is now `seatState === 'bot'`), don't restore them here. They'll need to use the mid-game join flow (Phase 30). For now, send a `kicked` or `game_ended` message so the client knows to return to menu. Use a simple approach: if `gamePlayer.seatState === 'bot'`, send `{ type: 'room_closed', reason: 'seat_taken_by_bot' }` and don't rejoin them. The client already handles room_closed by returning to menu.

  5. **Interaction with AFK timers**: If the player was also being tracked by the rolling/unlock timeout (they disconnected mid-phase), the AFK timer will still fire for them. Since they're now online and human-active, `autoRollUnresponsivePlayers` / `autoSkipUnresponsivePlayers` will handle them normally (they'll get a chance to act). No special handling needed — the existing 25s server timer is still running for all players in the phase.
  </action>
