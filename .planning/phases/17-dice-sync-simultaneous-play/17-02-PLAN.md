**Tech stack available:** partykit, partysocket, zustand, react, @react-three/fiber, @react-three/rapier
**Established patterns:**
- setRollResults() in gameStore: takes rolled values + positions + rotations, computes findAutoLocks, builds lock animations, processes AI rolls simultaneously
- DicePool fires onAllSettled(sortedValues, sortedPositions, sortedRotations) when all physics dice settle
- Scene handleAllSettled calls setRollResults with physics-determined values
- AI lock animations: fromScale 0 → toScale 1, fromPos at profile group position
- LockAnimation type: fromPos, toPos, fromRotation, value, delay, fromScale?, toScale?, playerId?
- Module-level getGameSocket for socket access
- pendingServerResults in store (from 17-01)

**Constraining decisions:**
- [Roadmap]: "Dice physics runs client-side (visual). Server determines roll RESULTS. Clients animate dice rolling, but actual values come from the server."
- [16-01]: Server generates dice for ALL players (online + bots), computes auto-locks, sends PlayerRollResult[] containing rolled[], newLocks[], poolSize, lockedDice for each player
- [16-01]: Server sends roll_results immediately after generating (fast — just random numbers)
- [05-03]: Scoring formula: penalties [1,0,1,1], roundScore = max(0, 8 - penalty sum)
- Results sorted ascending (required for lock animation correspondence)
- Server already sorts rolled values ascending in handleRollRequest

**Critical architecture: Timing barrier**
Server results arrive FAST (50-100ms). Physics dice take 1-3 seconds to settle. So typically: server results arrive first, physics settles second. The merge happens when BOTH are ready:
1. Server results arrive → stored in `pendingServerResults` (store field from 17-01)
2. Physics settle → Scene checks pendingServerResults:
   - If available: merge server values + physics positions → apply
   - If not available (rare): set a flag, wait for server results to arrive and trigger apply

  <action>
  Add new store action `applyOnlineRollResults` to gameStore. This is the online equivalent of `setRollResults` — it uses server-provided values instead of physics-detected values, but still uses physics positions for lock animations on the human player.

  **Signature:**
  ```typescript
  applyOnlineRollResults: (physicsPositions: [number, number, number][], physicsRotations: [number, number, number][]) => void;
  ```

  **Implementation logic:**
  1. Guard: `if (phase !== 'rolling') return` (same guard as setRollResults)
  2. Read `pendingServerResults` from store. If null, console.warn and return.
  3. Find local player's result: `pendingServerResults.find(r => r.playerId === onlinePlayerId)`. If not found, find by player index 0 as fallback.

  **For the local human player (player index 0):**
  - Use server's `rolled[]` as the dice values (NOT physics values)
  - Use server's `newLocks[]` as the lock results
  - Use `physicsPositions`/`physicsRotations` from physics settle for lock animation source positions
  - Build lockAnimations array using the same logic as setRollResults:
    - Map each lock to a physics die position by matching values (locksByValue consumption pattern)
    - fromPos = physics die position, toPos = [getSlotX(goalSlotIndex), DIE_SIZE/2, -3.77]
    - fromRotation = physics die rotation
    - Stagger delays: first 0, then prevDelay + 0.25-0.5s
  - Compute remainingDiceValues/positions/rotations by consuming locked values from rolled results (same pattern as setRollResults)
  - Update player: lockedDice = [...existing, ...newLocks], poolSize = server's poolSize

  **For other players (index 1+):**
  - Find their result in pendingServerResults by playerId matching
  - If they have newLocks: create lock animations using profile-emerge pattern (existing AI animation):
    - fromPos = [profileX, DIE_SIZE/2, rowZ] where profileX = getSlotX(0) - PROFILE_X_OFFSET, rowZ = -3.77 + playerIndex * 0.9
    - toPos = [getSlotX(lock.goalSlotIndex), DIE_SIZE/2, rowZ]
    - fromScale = 0, toScale = 1
    - playerId = player.id
    - Stagger delays
  - Update player: lockedDice = server's lockedDice, poolSize = server's poolSize
  - Do NOT generate local AI rolls — server already did this

  **Final state update:**
  - Set phase to 'locking'
  - Update roundState: rollResults (human's rolled values), rollNumber++, lastLockCount (human's newLocks.length), lockAnimations, animatingSlotIndices, aiLockAnimations, aiAnimatingSlotIndices, remainingDiceValues/positions/rotations
  - Clear pendingServerResults to null

  **Important:** The server's PlayerRollResult uses LockedDieSync (from protocol.ts), which has the same shape as LockedDie (from game.ts): `{ goalSlotIndex: number, value: number }`. They're compatible — just assign directly. Do NOT import or cast between types unnecessarily.

This is called from both `setPendingServerResults` and `setPhysicsSettledData` setters to handle either-arrives-first timing.
