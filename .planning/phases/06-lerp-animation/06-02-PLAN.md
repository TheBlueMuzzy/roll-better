**Layout reference:**
- GoalRow z = -4.67, PlayerRow z = -3.77
- getSlotX(i) = (i - 3.5) * 0.7, DIE_SIZE ≈ 0.659
- Rolling area: X from -2.8 to 2.8, Z from -1.7 (ROLLING_Z_MIN) to 4.5 (ROLLING_Z_MAX)
- Rolling area center: approximately [0, 0.33, 1.85]

**Current pool shrink behavior (the problem):**
When dice lock after a roll, `poolSize` decreases. DicePool detects `count < prevCount`:
- Bumps `generation.current++` (forces ALL dice to remount — BUG-001 fix)
- Recalculates spawn positions via `getSpawnPositions(count)` (centered grid)
- Sets `initialFace` from `remainingDiceValues` (correct face values)

Result: surviving dice snap from their physical resting positions to a centered grid.
The generation key bump is still needed (BUG-001 fix), but spawn positions should use the surviving dice's actual settled positions instead of a new grid.

**Current unlock flow (from 05-04):**
1. Phase is 'unlocking', player taps locked dice to select (selected = shrink 25%)
2. Player taps UNLOCK → handleConfirmUnlock → confirmUnlock(0) → immediate state change
3. setPhase('idle') — instant, no animation

**New unlock flow (this plan):**
1. Phase is 'unlocking', player taps locked dice to select (selected = translate UP)
2. Player taps UNLOCK
3. Animation plays (~1.3s total):
   - Lerp (0–0.5s): Die flies from player row to a clear spot in the pool area
   - Shake (0.5–0.9s): Die shakes with increasing intensity at the target spot
   - Split (0.9–1.3s): Die divides into two dice of the same value, moving apart
4. THEN confirmUnlock fires (state change)
5. THEN setPhase('idle')
6. DicePool re-renders — the two new dice appear at the split target positions

**AnimatingDie component (from 06-01):**
- Reusable for the lerp phase
- Props: fromPos, toPos, value, color, duration, onComplete
- Ease-in-out cubic + parabolic Y arc

**Established patterns:**
- useFrame ref mutation for animation (no React state for per-frame)
- Generation-counter keys in DicePool handle pool size changes
- Position capture: PhysicsDie reports settled position via body.translation()

  **Clear-spot algorithm (implement in a utility or inline in App.tsx):**
  Given existing pool dice positions (from `remainingDicePositions` in store), find a clear landing spot:
  1. Read `remainingDicePositions` from store — these are where pool dice currently sit
  2. For each unlock, need a spot with room for 2 dice side by side (the split result)
  3. Algorithm:
     - Generate candidate positions in the rolling area (grid of points within bounds)
     - Rolling area: X ∈ [-2.0, 2.0], Z ∈ [0.0, 3.5] (inner safe zone, away from walls)
     - For each candidate, compute minimum distance to any existing pool die
     - Pick the candidate with the maximum minimum distance (furthest from all dice)
     - splitTargets = [targetPos offset -DIE_SIZE*0.7, 0, 0] and [targetPos offset +DIE_SIZE*0.7, 0, 0] (horizontal split)
  4. For multiple unlocks, add previously computed targets to the "occupied" list so they don't overlap

  **MitosisDie.tsx (NEW component):**
  A three-phase animation component:

  Props: `fromPos, targetPos, splitTargets, value, color, onComplete`

  Internal state managed via refs (useFrame, no React state):
  - Phase 1 — LERP (0–0.5s): Single die flies from fromPos to targetPos
    - Ease-in-out cubic (same as AnimatingDie)
    - Parabolic Y arc for lob feel
    - Die A visible, Die B hidden
  - Phase 2 — SHAKE (0.5–0.9s): Die shakes at targetPos with increasing intensity
    - Oscillate X position: `sin(elapsed * freq) * amplitude`
    - amplitude ramps from 0.02 to 0.12 over the phase
    - freq = ~40 (fast vibration)
    - Die A visible at targetPos, Die B hidden
  - Phase 3 — SPLIT (0.9–1.3s): Die divides into two
    - Die A visible, Die B becomes visible
    - Both start at targetPos, ease-out to their respective splitTarget positions
    - Use ease-out cubic for the separation (fast start, decelerate)
    - Both dice show same value and color

  Structure: `<group>` containing two child `<group>` refs (dieGroupA, dieGroupB)
  - Each child has `<Die3D color={color} />` at scale DIE_SIZE
  - dieGroupB starts with `visible=false`, becomes `visible=true` at phase 3 start
  - Set rotation via `getRotationForFace(value)` on both

  Fire `onComplete` when phase 3 finishes (t >= 1.3s total).
