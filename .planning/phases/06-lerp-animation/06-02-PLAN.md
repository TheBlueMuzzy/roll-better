<objective>
Animate the unlock flow — selected locked dice depart from the player row, and bonus dice spawn from the goal row and fly to the pool area. Transforms the instant state change into a readable visual sequence.

Purpose: The unlock is a strategic decision with consequences (pool grows). The animation makes it feel weighty and helps the player track what just happened — their die returns, plus a bonus spawns from the goal.
Output: Unlock departure animation, bonus die spawn-from-goal animation, sequenced with confirmUnlock state change.
</objective>

**Layout reference:**
- GoalRow z = -4.67, PlayerRow z = -3.77
- getSlotX(i) = (i - 3.5) * 0.7, DIE_SIZE ≈ 0.659
- Rolling area center: approximately [0, 0.33, 1.85]
- DicePool spawn positions: getSpawnPositions(count) returns grid centered at rolling area

**Unlock flow (current — from 05-04):**
1. Phase is 'unlocking', player taps locked dice to select
2. Player taps UNLOCK (or SKIP) in HUD
3. App.tsx handleConfirmUnlock → store.confirmUnlock(0) → store updates immediately:
   - Removes selected locks from lockedDice
   - Adds pendingNewDice (each unlock → [value, value] = return die + bonus)
   - poolSize grows by 2 * numUnlocked
4. setPhase('idle') — immediate, no delay
5. DicePool re-renders with larger count, new dice get initialFace from pendingNewDice

**For animation, this plan changes the flow to:**
1. Phase is 'unlocking', user confirms unlock
2. Animation plays (~0.8s total):
   - Departure: selected locked dice shrink/fly away from player row (0.3s)
   - Spawn: bonus dice appear at goal row and fly to pool area (0.5s)
3. THEN confirmUnlock fires (state change)
4. THEN setPhase('idle')

**AnimatingDie component (from 06-01):**
- Reuse for bonus die flight from goal to pool
- Props: fromPos, toPos, value, color, duration, onComplete

**Established patterns:**
- useFrame ref mutation for animation (no React state for per-frame)
- Generation-counter keys in DicePool handle pool size changes

  **App.tsx — modify handleConfirmUnlock:**
  Currently it calls confirmUnlock(0) then setPhase('idle') immediately. Change to:

  1. Read current state BEFORE confirmUnlock: capture selectedForUnlock indices, lockedDice (for values), player.poolSize
  2. Compute animation data for each selected slot:
     - **Departure animation:** fromPos = `[getSlotX(slotIndex), DIE_SIZE/2, -3.77]` (player row position), toPos = `[0, DIE_SIZE/2, 1.85]` (pool center), duration = 0.3, type = 'departure'
     - **Spawn animation (bonus die):** fromPos = `[getSlotX(slotIndex), DIE_SIZE/2, -4.67]` (goal row position), toPos = `[0, DIE_SIZE/2, 1.85]` (pool center), duration = 0.5, type = 'spawn'
  3. Call `setUnlockAnimations(allAnimations)` — this triggers rendering
  4. Set a timeout: after 800ms (departure 0.3s + spawn 0.5s with overlap), call:
     - `confirmUnlock(0)` — NOW the state changes
     - `clearUnlockAnimations()`
     - `setPhase('idle')`

  **Timing detail:** Departure animations start immediately (0-0.3s). Spawn animations start after a 0.15s delay (0.15-0.65s) for visual stagger. Total animation window: ~0.7s, with 0.1s buffer = 0.8s timeout.

  Import getSlotX from GoalRow, DIE_SIZE from RollingArea. Use `useGameStore.getState()` for reading state inside the handler.
