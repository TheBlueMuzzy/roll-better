**A. Handle new message in useOnlineGame.ts:**

In the message handler switch statement, add a case for `"seat_state_changed"`:
```typescript
case "seat_state_changed":
  useGameStore.getState().updatePlayerSeatState(
    msg.playerId,
    msg.seatState,
    msg.seatIndex
  );
  break;
```

**B. Add updatePlayerSeatState action to gameStore.ts:**

Add a new action that updates a single player's seat state:
```typescript
updatePlayerSeatState: (playerId: string, seatState: SeatState, seatIndex: number) => {
  set(state => ({
    players: state.players.map(p =>
      p.id === playerId
        ? { ...p, seatState, seatIndex }
        : p
    ),
  }));
},
```

Import `SeatState` from protocol.ts if not already imported in gameStore.

**C. Handle the local player case:**
If the seat_state_changed is for the LOCAL player (playerId matches onlinePlayerId), the player was promoted to bot due to AFK. This means:
- The player's own seat is now bot-controlled
- They'll need to reclaim via mid-game join (Phase 30) — for now, just update the state
- Consider: should we show a more prominent "You've been replaced by a bot" message? For now, just sync the state — the UI task handles display.

**Important:** Do NOT add seatState to the protocol import in useOnlineGame.ts if it's not needed — the message object already has the field typed via ServerMessage union.

Add a simple state array for recent notifications:
```typescript
const [seatNotifications, setSeatNotifications] = useState<string[]>([]);
```

**B. Watch for seat state changes on OTHER players:**

Use a useEffect that watches `players` array. Compare previous seatState values to detect transitions. When a non-local player transitions:
- `human-active` -> `human-afk`: Show "{name} is on autopilot"
- `human-afk` -> `bot`: Show "Bot took over for {name}"
- `human-afk` -> `human-active`: Show "{name} is back"
- `bot` -> `human-active`: Show "{name} reclaimed their seat" (future — Phase 30)

Use a ref to track previous seat states and compare on each render.

Each notification auto-dismisses after 3 seconds. Show max 2 at a time (newest replaces oldest if overflow).

**C. Render notifications:**

Position below the existing HUD elements (near the bottom-center, above the roll button area). Style as small semi-transparent pill badges — white text on dark background with rounded corners, similar to the existing tip banner style but smaller.

**D. Handle LOCAL player autopilot:**

**Important:** Keep this lightweight. No modals, no blocking UI. Just informational pills that fade in/out. The seat state system is infrastructure — the full UX comes in later phases.
