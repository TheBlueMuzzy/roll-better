- Timer expires + connected + counter < 3 -> Human-AFK (1-beat autopilot, counter++)
- Timer expires + connected + counter = 3 -> Bot (full AI, seat claimable)
- Any manual action -> resets counter to 0, seatState back to human-active

- Bot = full AI with strategic decisions (same as existing bot difficulty system)

  <name>Task 1: Server autopilot counter + seat state machine</name>
  <files>party/server.ts</files>
  <action>
Wire autopilotCounter and seatState transitions into the existing AFK handling:

**A. Increment counter on autopilot fire:**
In `autoRollUnresponsivePlayers()` — for each AFK player processed:
- Increment `player.autopilotCounter`
- If `autopilotCounter >= 3`: call a new method `promoteToBotFromAFK(player)` that sets `player.seatState = 'bot'`, assigns a random difficulty (`player.difficulty = randomDifficulty()`), sets `player.isOnline = false` (so bot logic handles them going forward), and broadcasts the seat change
- If `autopilotCounter < 3` and `player.seatState !== 'human-afk'`: set `player.seatState = 'human-afk'` and broadcast the seat change

In `autoSkipUnresponsivePlayers()` — same logic for each AFK player processed:
- Increment `player.autopilotCounter`
- If `autopilotCounter >= 3`: call `promoteToBotFromAFK(player)`
- If `autopilotCounter < 3` and `player.seatState !== 'human-afk'`: set `player.seatState = 'human-afk'` and broadcast

**B. Reset counter on manual action:**
In `handleRollResult()` — after the validation checks pass (player found, values valid), before processing:
- If `player.autopilotCounter > 0`: reset `player.autopilotCounter = 0`
- If `player.seatState === 'human-afk'`: set `player.seatState = 'human-active'` and broadcast the seat change

In `handleUnlockRequest()` — after player found and validation passes:
- Same reset logic as above

In `handleSkipUnlock()` — after player found and validation passes:
- Same reset logic as above (skip IS a manual action — player chose to skip)

**C. Create `promoteToBotFromAFK(player)` method:**
- Set `player.seatState = 'bot'`
- Set `player.difficulty = randomDifficulty()`
- Set `player.isOnline = false` (server treats them as bot from now on)
- Remove their conn.id from `this.gameState.rollRequestedBy` (if present) — they're no longer an online player
- Log: `AFK escalation: ${player.name} promoted to Bot (${player.autopilotCounter} consecutive timeouts)`
- Broadcast `seat_state_changed` message (see Task 2)
- Do NOT remove from `this.players` map or disconnect their WebSocket — they may reconnect (Phase 29 handles this)

**D. Create helper `broadcastSeatStateChanged(player)`:**
- Broadcasts the new `seat_state_changed` message to all clients
- Also include the full player snapshot in the message for state recovery

**Important:** The 1-beat autopilot behavior (simplest legal move) is ALREADY what autoRollUnresponsivePlayers and autoSkipUnresponsivePlayers do — random dice for roll, AFK unlock rule for unlock. No change needed to the autopilot logic itself. The only new behavior is counter tracking and state transitions.

**Important:** When a player is promoted to bot, the CURRENT auto-action for that timeout still uses the simple 1-beat logic (it already fired). The bot difficulty kicks in starting from the NEXT phase — because `isOnline = false` means `checkAllRolled()` and `processAllUnlocks()` will treat them as a bot with `player.difficulty`.
