**PRD SS11 #8 seat states:**
| State | Meaning |
|-------|---------|
| Human-Active | Connected human playing this seat |
| Human-AFK | Connected but inactive, 1-beat autopilot fires at timer expiry |
| Bot | Full AI controls seat, seat is claimable by any human |

**This phase establishes the data model only.** Behavioral changes (AFK escalation = Phase 28, disconnect handoff = Phase 29) come later. For now:
- Online humans start as 'human-active'
- AI backfill bots start as 'bot'
- No state transitions happen yet (that's Phase 28+)

  <files>src/types/protocol.ts, party/server.ts</files>
  <action>
1. In `protocol.ts`, add the SeatState type:
   ```typescript
   export type SeatState = 'human-active' | 'human-afk' | 'bot';
   ```

2. In `protocol.ts`, update `PlayerSyncState` to include:
   - `seatState: SeatState`
   - `seatIndex: number` (0-based position in the game, stable across reconnects)

3. In `protocol.ts`, update `RoomPlayer` to include:
   - `seatIndex: number` (assigned on join, shown in lobby)

4. In `server.ts`, update `ServerPlayerState` to include:
   - `seatState: SeatState` (import from protocol.ts)
   - `seatIndex: number`
   - `autopilotCounter: number` (consecutive AFK count, starts at 0)

5. In `server.ts` `handleStartGame()`, when building gamePlayers:
   - Online players: `seatState: 'human-active'`, `seatIndex: index`, `autopilotCounter: 0`
   - Bot players: `seatState: 'bot'`, `seatIndex: index`, `autopilotCounter: 0`
   - seatIndex = position in the gamePlayers array (0, 1, 2, ...)

6. In `server.ts` `handleRestartGame()`, same pattern for seat initialization.

7. In `server.ts` `handleJoin()`, assign `seatIndex` to RoomPlayer based on current player count (this is lobby ordering). Update the RoomPlayer creation to include seatIndex.

8. In `server.ts` `buildPlayerSnapshot()`, include `seatState` and `seatIndex` in the returned PlayerSyncState objects.
