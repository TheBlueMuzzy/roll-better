<objective>
Add persistent player identity (localStorage) and server-side seat mapping so the server can track which persistent player owns which seat across browser sessions.

Purpose: Foundation for all drop-in/drop-out features (phases 28-34). Without persistent identity and seat mapping, the server can't reconnect returning players to their seats.
Output: Persistent player ID generated on first visit, sent in join messages, stored in server seat map.
</objective>

**Current identity system:**
- `partyClient.ts:getStableClientId()` generates 16-char ID in sessionStorage (key: `rb-client-id`)
- `PartySocket({ id: getStableClientId() })` makes conn.id = this session-scoped ID
- Server uses conn.id as the player identity everywhere
- Reconnection works via conn.id matching in onConnect — only within same tab session

**Design decision (from PRD SS11 #8):**
- Keep sessionStorage conn.id for PartySocket WebSocket reconnection (within-session)
- ADD localStorage-based persistent player ID for cross-session identity
- Server stores both: conn.id for connection management, persistentId for seat ownership
- If localStorage is cleared, player is treated as brand new (accepted edge case per PRD)

   - `persistentIdToConnId: Map<string, string>` — maps persistent player ID to current conn.id

4. In `handleJoin()`:
   - Extract `persistentId` from the parsed join message (cast parsed to access it)
   - Store it in the RoomPlayer entry: `persistentId` field
   - Add to mapping: `this.persistentIdToConnId.set(persistentId, conn.id)`
   - Log: `[JOIN] persistentId=${persistentId.slice(0,8)} connId=${conn.id.slice(0,8)}`

5. In `handleStartGame()` and `handleRestartGame()`, when building `gamePlayers` array for online players, copy `persistentId` from the RoomPlayer into ServerPlayerState.

6. In `onConnect()` connected ack, include `persistentId` from the join data. Since at onConnect time we don't have the persistent ID yet (it comes in the join message), send an empty string for now — the real persistent ID confirmation happens when room_state is sent after join. Update the ConnectedMessage to use `persistentId: ""` placeholder.

7. In `removePlayer()`, when a player disconnects, do NOT remove them from `persistentIdToConnId` — the mapping persists so returning players can be identified. Only clear it when the room dissolves.

8. Bot players get persistentId = their bot id (e.g., `"bot-0"`, `"bot-1"`).

9. In `buildPlayerSnapshot()`, do NOT include persistentId in PlayerSyncState yet (that's Plan 27-02).
